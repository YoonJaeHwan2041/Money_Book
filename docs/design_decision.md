# 🧠 설계 의사결정 (Design Decisions)

이 문서는 MoneyBook 프로젝트의 주요 아키텍처 및 데이터베이스 설계 결정 사항을 기록합니다.

---

## 1. 데이터베이스 모델링: 통합 트랜잭션 테이블
- **결정**: 일반 지출, 수입, 고정 지출, 정산 지출을 `Transaction` 단일 테이블에서 관리.
- **이유**:
    - **통계 효율성**: 전체 지출 합계 등 통계 산출 시 다중 조인(Join) 없이 단일 쿼리로 처리 가능.
    - **데이터 일관성**: 모든 금융 흐름의 'Single Source of Truth' 확보.
    - **상태 관리**: `is_confirmed` 필드를 통해 예정 지출과 실제 지출을 유연하게 전환 가능.

## 2. 데이터 타입 선택: Unix Timestamp (Long)
- **결정**: 날짜 및 시간 데이터를 `Long` 타입으로 저장.
- **이유**:
    - **성능**: 숫자형 정렬 및 비교 연산이 문자열보다 압도적으로 빠름.
    - **정확성**: 타임존(Timezone) 문제 방지 및 앱 로직에서의 자유로운 포맷팅 지원.

## 3. 리소스 관리: 식별자(String Key) 전략
- **결정**: 카테고리 아이콘을 `icon_key` 문자열로 관리.
- **이유**:
    - **확장성**: 앱 내장 리소스(`ic_...`)와 사용자 커스텀 파일 경로를 하나의 필드에서 구분하여 처리 가능.
    - **유연성**: DB 수정 없이 코드 레벨에서 아이콘 매핑 로직 변경 가능.
